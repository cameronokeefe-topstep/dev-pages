# Jekyll Websites — Overview

## What Jekyll is (in one paragraph)
Jekyll is a **static site generator**: you write content in **Markdown/HTML**, configure site behavior with **YAML**, and Jekyll builds a folder of **static files** (HTML/CSS/JS) you can host anywhere (GitHub Pages, Netlify, S3, etc.). It shines for blogs, docs, and marketing sites that don’t need a dynamic backend.

---

## Mental Model
**Inputs**
- Markdown posts/pages
- Layouts and includes (Liquid templates)
- Data files (YAML/JSON/CSV)
- Config (`_config.yml`)
- Assets (`assets/`)

**Build**
- Jekyll processes Markdown + Liquid + front matter
- Generates static HTML into `_site/`

**Output**
- `_site/` is the deployable website

# Key Concepts

## 1) Front Matter (YAML)

Front matter is YAML at the top of a Markdown/HTML file that sets metadata and controls rendering.

```yaml
---
layout: post
title: "Hello Jekyll"
date: 2026-02-24
permalink: /blog/hello/
categories: [news]
tags: [jekyll, liquid]
---
```

## 2) Layouts (_layouts)

Layouts wrap content. A layout must include `{{ content }}` where the page/post body should appear.

```html
<!-- _layouts/default.html -->
<!doctype html>
<html lang="en">
  <head>{% include head.html %}</head>
  <body>
    {% include header.html %}
    <main>{{ content }}</main>
    {% include footer.html %}
  </body>
</html>
```

## 3) Includes (_includes)

Reusable partials: nav, head tags, footer, etc.

```liquid
{% include header.html %}
```

## 4) Liquid (templating language)

Jekyll uses Liquid for variables, loops, conditionals, and filters.

```bash
bundle install
```

## Build


# _config.yml Cheatsheet

```yaml
title: My Site
description: Notes and articles
url: "https://example.com"     # production base
baseurl: ""                    # if hosted at /subpath, set "/subpath"

markdown: kramdown
permalink: /blog/:title/       # or /:categories/:year/:month/:day/:title/

timezone: America/New_York

collections:
  pages:
    output: true

defaults:
  - scope:
      path: ""
      type: posts
    values:
      layout: post

exclude:
  - node_modules
  - vendor
  - Gemfile
  - Gemfile.lock
```

**Tip:** `baseurl` matters a lot on GitHub Pages project sites.

# Posts Cheatsheet

## Post file naming

```text
_posts/YYYY-MM-DD-title.md
```

## Post example

```markdown
---
layout: post
title: "My First Post"
---
This is my post body.
```

## Loop all posts

```liquid
<ul>
  {% for post in site.posts %}
    <li>
      <a href="{{ post.url | relative_url }}">{{ post.title }}</a>
      <small>{{ post.date | date: "%b %d, %Y" }}</small>
    </li>
  {% endfor %}
</ul>
```

## Drafts

Put drafts in `_drafts/` without a date prefix.

Serve them with:

```bash
bundle exec jekyll serve --drafts
```

# Pages Cheatsheet

## A page at /about/

Create `about.md`:

```markdown
---
layout: page
title: About
permalink: /about/
---
Hello!
```

## Page listing example

```liquid
{% assign pages = site.pages | where_exp: "p", "p.title" %}
<ul>
  {% for p in pages %}
    <li><a href="{{ p.url | relative_url }}">{{ p.title }}</a></li>
  {% endfor %}
</ul>
```

# Liquid Basics Cheatsheet

## Output a variable

```liquid
{{ page.title }}
{{ site.title }}
```

## If / else

```liquid
{% if page.layout == "post" %}
  <p>Reading a post</p>
{% else %}
  <p>Not a post</p>
{% endif %}
```

## For loop

```liquid
{% for tag in page.tags %}
  <span>{{ tag }}</span>
{% endfor %}
```

## Common filters

```liquid
{{ "hello world" | upcase }}
{{ page.date | date: "%Y-%m-%d" }}
{{ "/assets/css/site.css" | relative_url }}
{{ site.url | append: page.url }}
```

## Assign / capture

```liquid
{% assign featured = site.posts | first %}
{{ featured.title }}

{% capture hero %}
  <h1>{{ page.title }}</h1>
{% endcapture %}
{{ hero }}
```

# URLs & Linking (the “don’t break in prod” section)

## Use relative_url and absolute_url

```html
<a href="{{ '/about/' | relative_url }}">About</a>
<link rel="stylesheet" href="{{ '/assets/css/site.css' | relative_url }}">

<a href="{{ '/about/' | absolute_url }}">Absolute About</a>
```

### Rule of thumb:

- Use `relative_url` for internal paths (safe with baseurl)
- Use `absolute_url` for canonical URLs, RSS, SEO tags

# Assets & Styling Cheatsheet

## Add a stylesheet

```html
<link rel="stylesheet" href="{{ '/assets/css/site.css' | relative_url }}">
```


## Sass support (optional)

If you use Sass:

- Put `.scss` in `_sass/` (partials)
- Put a main `.scss` in `assets/css/`

Example main file:

```scss
---
---
@import "base";
@import "layout";
```

# GitHub Pages Notes (practical)

GitHub Pages supports Jekyll builds, but not all plugins (unless you build elsewhere and push `_site/`).

If you need unrestricted plugins, build on CI (GitHub Actions) and deploy the static output.

# Debugging Cheatsheet

## See config values

```liquid
<pre>{{ site | jsonify }}</pre>
```

## Common gotchas

- Missing front matter → file won’t be processed as expected
- baseurl mismatch → broken links/assets on GitHub Pages project sites
- Wrong permalink → 404s
- Liquid syntax errors → build fails; check terminal output line numbers
- Windows line endings can sometimes cause weirdness in scripts/config

# Checklist: Starting a New Jekyll Site

- Create `_config.yml` with url and baseurl
- Add `_layouts/default.html` with `{{ content }}`
- Add `_includes/head.html`, header.html, footer.html
- Create `index.md`
- Add first post in `_posts/`
- Serve locally with `bundle exec jekyll serve --livereload`
- Deploy `_site/` or let GitHub Pages build (depending on plugin needs)
